## The Brief

The goal was to build [js17.dev](https://js17.dev) — a personal professional website for a Senior AI-Augmented Fullstack Systems Engineer — from an empty directory to a live, DNS-verified, SSL-secured, fully tested production deployment.

**Constraints:**
- Vercel free tier
- No external CMS — blog posts as MDX files in the repo
- No static OG images — dynamic generation via Next.js `ImageResponse`
- Emails via Resend (not a third-party form service)
- Cal.com for scheduling (not Calendly, not a link to Google Calendar)
- 84 automated checks across 9 test suites before declaring success

## What I Brought to the Session

Let me be direct about this. The AI wrote the code. I directed everything that the code needed to be.

### 1. The Architecture Document

Before writing a single prompt, I produced a six-phase implementation plan covering:

- Framework selection with justification
- File structure down to individual component names
- Render strategy per route (SSG, ISR, CSR, Edge)
- Environment variable schema
- Performance targets
- Verification criteria

This is not a task you can delegate to an AI. Producing this document requires knowing what a production Next.js App Router project looks like at scale, understanding the tradeoffs between MDX solutions, knowing why `next-mdx-remote@6` behaves differently from `@next/mdx`, and having opinions about component architecture before any code exists.

**Skills required:** Systems architecture, Next.js internals, performance engineering, product thinking.

### 2. Technology Triage

Several decisions involved choosing between options with non-obvious tradeoffs:

- **MDX solution:** `@next/mdx` vs `next-mdx-remote` vs `contentlayer`. I chose `next-mdx-remote` for the `compileMDX` RSC API, knowing it works cleanly with the App Router.
- **OG images:** Static PNG vs `ImageResponse` Edge function. I chose dynamic generation — one file handles all blog posts without manual image creation.
- **Email provider:** SendGrid vs Resend vs Postmark. I chose Resend for its Next.js-native SDK and generous free tier.
- **Scheduling:** Calendly vs Cal.com. I chose Cal.com for the inline embed and Google Calendar sync.

**Skills required:** Vendor evaluation, API design pattern recognition, cost/complexity analysis.

### 3. Security Discipline

Throughout the session, API keys were provided directly when needed and never written to files. When Claude suggested approaches that would have exposed credentials, I corrected the direction. The `.gitignore` was verified. The Resend API key was set only as a Vercel environment variable, never in the codebase.

When the build failed with **CVE-2026-0969** in `next-mdx-remote@4`, I immediately recognized what a CVE flag in a CI/CD pipeline means and directed the upgrade to v6 rather than ignoring the warning.

**Skills required:** OWASP awareness, secrets management, dependency auditing.

### 4. Precise Information at the Right Moment

Every external service required exact credentials provided at the right step:

- GitHub personal access token (PAT) with `public_repo` scope
- Resend API key scoped to `js17.dev`
- Recipient email address
- Cal.com username (`jeroham-sanchez-gfvgbp`) — not guessable, not skippable
- Cloudflare API token with `Zone:DNS:Edit` permission (after identifying that the first token had an IP restriction)

The quality and precision of this information directly determined whether each integration worked on the first attempt or required debugging. Vague inputs produce vague outputs. Every key was provided in the correct format, at the correct step.

**Skills required:** API credential management, service integration knowledge, systematic onboarding.

### 5. DNS and Infrastructure Diagnosis

This was where domain expertise separated success from a several-hour detour.

When `https://js17.dev` returned 200 but served the wrong content, I diagnosed three simultaneous issues:

1. **Two stale GoDaddy A records** (`13.248.243.5`, `76.223.105.230`) competing with the Vercel A record
2. **Cloudflare proxy enabled** (orange cloud) on both the A record and the `www` CNAME — which intercepts TLS and breaks Vercel's SSL provisioning
3. **SSL certificate not auto-provisioning** — Vercel requires `vercel certs issue` when DNS was recently migrated

Each of these is a real infrastructure failure mode that requires knowing how Cloudflare's proxy layer interacts with origin SSL, what Vercel's certificate provisioning expects, and how to use the Cloudflare API programmatically.

```bash
# The three fixes, in sequence:
# 1. Delete competing A records via Cloudflare API
# 2. Set proxied: false on A + CNAME records
# 3. Manually trigger cert issuance
vercel certs issue js17.dev www.js17.dev
```

**Skills required:** DNS architecture, Cloudflare configuration, SSL/TLS certificate lifecycle, Vercel deployment internals.

## The Challenges

### Challenge 1: OG Image — Edge Runtime vs Node.js

The blog `[slug]/opengraph-image.tsx` initially imported `@/lib/mdx` to read the post title. This worked in development but failed at build time: `@vercel/og` runs in the Edge runtime, which cannot access Node.js `fs` or `path` modules.

The fix required understanding *why* the constraint exists (Edge functions run in a V8 isolate, not a Node.js process) and designing around it: derive the post title directly from the URL slug parameter, avoiding any file system access entirely.

```typescript
// Before: import from mdx.ts → uses fs/path → fails in Edge runtime
// After: derive title from slug parameter — zero Node.js APIs
const slugTitle = params.slug
  .split("-")
  .map(w => w.charAt(0).toUpperCase() + w.slice(1))
  .join(" ")
```

### Challenge 2: CVE-2026-0969 Blocking Production Deploy

Vercel's build pipeline flagged a critical vulnerability in `next-mdx-remote@4` and refused to deploy. Upgrading to v6 required knowing that the `compileMDX` API from `next-mdx-remote/rsc` remained compatible — not a guaranteed assumption when jumping major versions.

### Challenge 3: Resend Build-Time Initialization

The Resend client was instantiated at module level:

```typescript
// This throws at build time if RESEND_API_KEY is not set
export const resend = new Resend(process.env.RESEND_API_KEY)
```

Next.js evaluates API route modules during the build step, where environment variables are not yet injected. The fix was a lazy initialization pattern — instantiate only on first request, never at import time.

### Challenge 4: Vercel Deployment Protection (401 on Preview URLs)

The verification script returned 401 on all routes against the preview URL. This required recognizing that Vercel's deployment protection is applied to preview deployments by default, and that the stable project alias (`js17devcv.vercel.app`) bypasses it — then running the full 84-check suite there before the canonical domain was live.

### Challenge 5: Three-Way DNS Conflict

Described above. The root cause was a GoDaddy-to-Cloudflare migration that left legacy A records in place, combined with Cloudflare's default proxy-on behavior overriding the new records.

## What AI-Augmented Engineering Actually Means

This project is a working definition of the term.

AI-augmented engineering is not "AI writes code, human reviews it." That is AI-assisted copy-paste, and it produces exactly the quality you would expect.

AI-augmented engineering is **the human operating as the architect, the decision-maker, the quality gate, and the infrastructure owner** — using AI as a force multiplier that eliminates the gap between design and implementation.

The multiplier only works when the human brings:

- An architecture that is worth implementing
- Technology choices that are actually correct for the constraints
- Security discipline that doesn't get eroded under time pressure
- Domain knowledge for the layers that AI cannot access (DNS panels, API dashboards, production credentials)
- A verification standard rigorous enough to catch the gap between "it looks like it works" and "84 checks pass against the live production URL"

Remove any one of those, and the multiplier disappears. What remains is a faster way to produce mediocre software.
